For the last few months I've been tinkering with improving this site for AI agents *as consumers* of its content, but after reading [Dachary Carey's article on Astro's removal of `llms.txt`](https://dacharycarey.com/2026/05/04/astro-removed-llms-txt/) I went down a bit of a rabbit hole which led me to [Joost de Valk's "agent-ready" piece](https://joost.blog/agent-ready/) and a bunch of smallish changes outlined below.

<Callout>
Although I deploy this site to Vercel, it's important to me that it remains a statically-built site which doesn't **depend** on Vercel's build pipeline or features. I covered this in my [last article](/writing/speeding-up-astro-builds) but the main point is this: **I want my static build to work on any server**.
</Callout>

## Existing "agent friendly" features

This site has always had stuff to help machines understand its content. It has sane, predictable URLs, seo-friendly meta tags and `<link rel="canonical">` in the head of every page, a sitemap generated by `@astrojs/sitemap` and three RSS feeds advertised on every page. The markup is semantic and includes a sprinkling of [microformats](https://microformats.org/wiki/microformats2) and [schema.org](https://schema.org/docs/documents.html) markup in appropriate places. All pages have sensible OG Data and [auto-generated OG images](/notes/nicer-cover-images/). I also publish [machine-readable records over AT Protocol](/notes/2026-06-04-this-site-on-atproto/).

Over the last year I've also added a few things *specifically* aimed at AI tools...

### Markdown versions of my content

Every article and note has a static `.md` twin generated by `[...slug].md.ts` endpoints, so appending `.md` to any URL will return a markdown version of the page ([try it for this article](/writing/making-this-astro-site-agent-friendly.md)). I also [include "copy / open as Markdown" buttons](/notes/adding-share-features/) at the end of every post so it's a little easier for humans to paste my content into chat interfaces and the like.

### `llms.txt`

I generate a [`llms.txt`](https://llmstxt.org/) at build time which includes:

- A brief intro about who I am and what the site is.
- Links to my socials, avatar image and the like
- What I'm doing now, pulled from my [now page](/now).
- A list of all the [articles](/writing) on this site.
- A list of all the [notes](/notes) on this site.
- A list of [other pages](/notes/new-meta-pages), including the [styleguide](/styleguide) and some important external pages.

You can see it [here](/llms.txt).

### JSON-LD

I include a full [JSON-LD](https://json-ld.org/) `@graph` of structured data in every page which includes `Person`, `Organization` and `WebSite`, plus `BlogPosting` for content pages.

```html
<script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@graph": [
      { "@type": "Person", "@id": "https://danny.is/#person" },
      { "@type": "WebSite", "@id": "https://danny.is/#website" },
      { "@type": "BlogPosting", "headline": "…", "datePublished": "…" }
    ]
  }
</script>
```

## Auditing for Agent Friendliness

Dachary Carey's [`afdocs` tool](https://www.npmjs.com/package/afdocs) checks against the spec at [agentdocsspec.com](https://agentdocsspec.com/spec/), which is primarily designed to ensure *content* is agent-readable and navigable. Cloudflare's **[isitagentready.com](https://isitagentready.com/)** is more focussed on conformity with emerging agent _discovery_ standards. Running `bunx afdocs check http://danny.is --verbose` yielded [these warnings & failures](https://github.com/dannysmith/dannyis-astro/issues/127#issue-4635722651), while isitagentready.com gave me a pretty shit score and recommended a bunch of potential improvements.

As with most automated audit tools, these are blunt instruments with very little context about this website. So the goal here isn't improving scores, it's using the data to actually improve the site.

## Better Agent-Friendliness

Here's what I actually did:

### 1. Update `robots.txt`

I updated my `robots.txt` to use the [Content Signals](https://contentsignals.org/) convention:

```
User-Agent: *
Content-Signal: search=yes, ai-input=yes, ai-train=no
Allow: /
```

This is a preference signal, so the AI firms will probably ignore it.

### 2. Add pointers to `llms.txt` to every page

My `llms.txt` exists purely to tell AI agents how to get what they need from this site, so it's super important that they can **actually find it**. The HTML for an article looks something like this:

```html
<head>
  ...
</head>
<body>
  <main>
    <article class="h-entry" itemscope itemtype="https://schema.org/Article" >
      ...
    </article>
  </main>
</body>
```

Which means any agent using a fetch tool (like Claude's `WebFetch`) that returns markdown is extremely unlikely to ever see anything outside of `<main>`. If we want every fetch to include a pointer to `llms.txt`, it's not enough to include pointers in our `<head>`.

I now insert a `LLMDiscoveryNote.astro` component immediately after the opening `<main>` on every page:

```astro
<p class="sr-only">
  For AI agents and crawlers: a structured index of this site is available at
  <a href="https://danny.is/llms.txt">https://danny.is/llms.txt</a>.
</p>
```

<Callout>
The *"For AI agents and crawlers"* isn't just there for the agents and crawlers. Since this'll be read to anyone using a screen reader, it's also there so they know this isn't for them. If anyone knows of a reliable way to hide this from screen readers and not from crawlers & agents please tell me.
</Callout>

We need to do something similar for the markdown versions of our content, so each now includes this near the top too:

```md
> For the complete site index, see [llms.txt](https://danny.is/llms.txt)
```

### 3. Serve markdown where a UA asks for it

Since user agents can send an [`Accept` header](https://developer.mozilla.org/en-US/docs/Web/HTTP/Content_negotiation) with their requests and we already have markdown versions of our content, we should probably serve markdown to anyone requesting plain ol' `/writing/<slug>` URLs with an `Accept: text/markdown` header. We can do this for both notes and articles with a little addition to our vercel config:

```jsonc
{
  "src": "^/writing/([^/]+)/?$",
  "has": [{ "type": "header", "key": "accept", "value": ".*text/markdown.*" }],
  "dest": "/writing/$1.md",
  "headers": { "content-type": "text/markdown; charset=utf-8" }
}
```

Browsers never send `Accept: text/markdown` so they're unaffected by this, but we need to make sure "negotiable URLs" like this carry `Vary: Accept` so CDNs don't accidentally serve markdown when they shouldn't.

While we're here, we can also respond to `Accept: text/markdown` requests to `danny.is` with the contents of our `llms.txt` rather than our homepage HTML.

### 4. Advertise the index Even More

Since llms.txt is so important, let's also add a `Link` header to every response and a `<link>` tag to every HTML document:

```
Link: </llms.txt>; rel="describedby"
```

```html
<link rel="describedby" type="text/markdown" href="/llms.txt">
```

### 5. Make it easier to discover markdown versions

Our `llms.txt` currently includes a big old list of markdown links to all our content, but the URLs in that list point at the *HTML* versions. It'd make much more sense to use the `.md` URLs in this context.

While we're here, we should make sure our content pages include a `<link rel="alternate" type="text/markdown" href="/writing/<slug>.md">` and all our `*.md` endpoints return a `text/markdown` Content-Type.

### 6. Turn meta-refresh redirects into real HTTP redirects

I use redirects in two places on this site:

1. Those hard-coded in `redirects.ts` which forward URLs like danny.is/meeting to external services.
2. Articles with a `redirectURL` frontmatter property which forward to external content pieces I've written.

I use [Astro redirects](https://docs.astro.build/en/guides/routing/#configured-redirects) for the former and meta-refresh pages for the latter, though since this is a statically generated site they all end up as HTML meta-refresh pages in the generated `/dist`.

We can improve on this by having our `astro build` *also* generate a host-agnostic `dist/redirects.json` containing **all** of these, in this kinda form:

```json
{
  "version": 1,
  "redirects": [{ "source": "/meeting", "destination": "https://cal.com/dannysmith", "status": 302 }]
}
```

Then as part of our Vercel-specific deployment action we can use this to generate an appropriate [Build Output API](https://vercel.com/docs/build-output-api) config which returns proper `302` redirects for these routes. The meta-refresh pages stay in the static build for the sake of portability, while `dist/redirects.json` exists in every static build as the source of truth about how server-side redirects should work.

## So where are we now?

These changes made quite a difference to our earlier metrics:

```bash
→ bunx afdocs check https://danny.is --verbose
...
16 passed, 3 warnings, 3 failed, 1 skipped (23 total)
```

All of the warnings and failures are "false positives" as far as I'm concerned (if you're interested in the details go ahead and run it yourself).

The isitagentready audit jumped from a score of ~20 to 43 overall, with all the things I care about being "green" ([check it out yourself](https://isitagentready.com/danny.is)).

![isitagentready.com showing danny.is at a score of 43, with the content and discovery checks I care about passing](../../assets/articles/2026-06-19-isagentready.png)

Most of the "failures" here have to do with API, Auth, MCP & Skill Discovery. Not only are these extremely new/immature standards, they're also not meant for simple *content* sites like this.

## Wrapping up

None of this was much work, and the nice thing is that it all degrades gracefully. An agent that ignores every signal still gets clean, semantic HTML; a dumb static host with no Vercel config still serves the whole site correctly. That's really the point — I'm not chasing a score, I'm making sure that whether you're a human, a crawler, or some agent poking around on someone's behalf, you can get at what's here without fighting the site. I'll keep half an eye on the newer stuff — MCP, agent cards and friends — but I'll only bother with them if and when they actually make sense for a little content site like this one.
